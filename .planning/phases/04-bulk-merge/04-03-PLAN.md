**Part A: Create `src/commands/merge.ts`**

Export `mergeCommand(file: string, options: MergeOptions): Promise<void>`.

Follow the four-phase pipeline pattern from `src/commands/update.ts`:

**Preamble:**
- `setVerbose(options.verbose)`
- `const network = resolveNetwork(options.network)`
- Log section header: "Geo Publish - Merge" with file, network, dry-run key-values

**PHASE 1: VALIDATE**
1. Validate file exists (`fs.existsSync`)
2. Parse template: `const { pairs, spaceId, operationType, errors } = parseMergeTemplate(filePath)`
3. If errors, log and `process.exit(1)`
4. If no pairs, error and exit
5. Warn if operationType is not 'MERGE' (same pattern as update.ts checking UPDATE)
6. Collect all unique entity names from pairs (both keeper and merger names) into a single list
7. Resolve all names via `searchEntitiesByNames(allNames, spaceId, network)`
8. Check for unresolved names -- hard error if any name can't be resolved:
   ```
   Cannot resolve N entity name(s):
     - name1
     - name2
   All entities must exist in Geo before merging.
   ```
9. Build resolvedEntities map: `Map<normalizedName, { id: string; name: string }>`
10. Log success: "All N names resolved"

**PHASE 2: DIFF**
1. For each pair in file order:
   a. Look up keeper and merger IDs from resolvedEntities map
   b. Fetch full entity details for both: `fetchEntityDetails(keeperId, spaceId, network)` and `fetchEntityDetails(mergerId, spaceId, network)`
   c. Hard-error if either fetch returns null (entity disappeared between validation and diff)
   d. Call `computeMergePairDiff(keeperDetails, mergerDetails)` from merge-diff.ts
   e. Collect the diff
   f. Log progress: `Processing pair N/M: KeeperName <- MergerName`
2. Compute `MergeSummary` by aggregating across all diffs:
   ```typescript
   const summary: MergeSummary = {
     totalPairs: diffs.length,
     propertiesTransferred: diffs.reduce((sum, d) => sum + d.propertiesToTransfer.length, 0),
     conflictsDetected: diffs.reduce((sum, d) => sum + d.conflicts.length, 0),
     relationsRepointed: diffs.reduce((sum, d) => sum + d.relationsToRepoint.length, 0),
     relationsSkipped: diffs.reduce((sum, d) => sum + d.relationsSkipped.length, 0),
     typesTransferred: diffs.reduce((sum, d) => sum + d.typesToTransfer.length, 0),
     mergersDeleted: diffs.length,
   };
   ```

**DRY-RUN GATE** (stops here if --dry-run):
- Call `printMergeDiffOutput(diffs, summary, { verbose: options.verbose })`
- Generate and save report: `generateMergeReport(diffs, summary, spaceId, network, true)` + `saveOperationReport()`
- Log "Dry run complete -- no changes were made." and exit

**PHASE 3: CONFIRM**
- Call `printMergeDiffOutput(diffs, summary, { verbose: options.verbose })`
- Check if any work to do (all transfers/re-points zero across all pairs -> "No changes to merge")
- If not `options.yes`: `confirmAction('Merge these entities? This action cannot be undone.')`

**PHASE 4: PUBLISH**
1. Validate PRIVATE_KEY (same pattern as update.ts: check env var + validatePrivateKey)
2. Save pre-merge snapshot of ALL keeper and merger entity details to `.snapshots/`:
   ```typescript
   const snapshotData = diffs.map(d => ({
     keeper: keeperDetailsMap.get(d.keeperId),
     merger: mergerDetailsMap.get(d.mergerId),
   }));
   ```
   Use same snapshot function pattern as delete.ts's `saveSnapshot()`. Filename: `merge-snapshot-{timestamp}.json`
3. **Per-pair atomic publishing** (MERGE-06, CONTEXT.md locked):
   - For each diff in file order:
     a. Build ops: `const ops = buildMergeOps(diff, diff.keeperId)`
     b. Build OperationsBatch with zeroed BatchSummary adapter (same pattern as update/delete)
     c. Build Metadata: `{ spaceId, spaceType: 'Personal' }`
     d. Call `publishToGeo(batch, metadata, privateKey, publishOptions)`
     e. Log result per pair: success/failure with transaction hash
     f. If publish fails, log error but continue to next pair (do NOT abort entire run -- pairs already published are already committed). Track success/failure per pair.
   - **CRITICAL for multi-way merges:** After publishing a pair, if the NEXT pair has the SAME keeper, re-fetch the keeper's details via `fetchEntityDetails()` before computing the next diff. This is handled naturally since diffs are already computed in Phase 2. However, for multi-way merges to be fully correct, re-fetching would be needed. Since RESEARCH.md recommends re-fetching for correctness but diffs are pre-computed, add a comment noting this limitation: "// NOTE: Multi-way merges use pre-computed diffs. For maximum correctness with many merger-to-one-keeper scenarios, consider re-fetching keeper state between publishes. Current approach is sufficient for typical use (tens of pairs)."
4. Generate and save report for live run
5. Print final summary via `printMergeSummary()`
6. Log completion section with transaction links

**Error handling:**
- Wrap entire pipeline in try/catch (same as update.ts pattern)
- On unexpected error: log, show snapshot path if available, exit 1

    });
