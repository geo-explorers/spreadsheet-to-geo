  <action>
Create `src/processors/merge-diff.ts` with the following functions:

**1. `buildKeeperRelationSet(keeper: EntityDetails): Set<string>`**

Build a dedup set from keeper's existing relations for O(1) duplicate checking:
- For each outgoing relation: add key `out:${rel.toEntity.id}:${rel.typeId}`
- For each incoming backlink: add key `in:${bl.fromEntity.id}:${bl.typeId}`
- Return the Set

**2. `extractTypedValue(value: EntityDetails['values'][0]): { typedValue: TypedValue; humanReadable: string } | null`**

Extract the non-null field from an EntityDetails value entry and reconstruct a TypedValue for Graph.updateEntity:
- Check each field in order: `text`, `boolean`, `integer`, `float`, `date`, `time`, `datetime`, `point`, `schedule`
- For the first non-null field found, return the corresponding TypedValue:
  - `text` non-null: `{ type: 'TEXT', value: text }`, humanReadable = text
  - `boolean` non-null: `{ type: 'CHECKBOX', value: String(boolean) }`, humanReadable = String(boolean)
  - `integer` non-null: `{ type: 'NUMBER', value: String(integer) }`, humanReadable = String(integer)
  - `float` non-null: `{ type: 'NUMBER', value: String(float) }`, humanReadable = String(float)
  - `date` non-null: `{ type: 'TIME', value: date }`, humanReadable = date
  - `time` non-null: `{ type: 'TIME', value: time }`, humanReadable = time
  - `datetime` non-null: `{ type: 'TIME', value: datetime }`, humanReadable = datetime
  - `point` non-null: `{ type: 'POINT', value: point }`, humanReadable = point
  - `schedule` non-null: return null (skip schedule properties -- not transferable as simple TypedValue)
- If all fields are null, return null

**3. `computeMergePairDiff(keeper: EntityDetails, merger: EntityDetails): MergePairDiff`**

Core diff engine. Compare keeper and merger to compute what to transfer:

**Property comparison** (CONTEXT.md locked decisions):
- Build a Map of keeper's properties: `Map<propertyId, humanReadableValue>` from keeper.values
- For each property in merger.values:
  - **SKIP if propertyId === SystemIds.NAME_PROPERTY** -- keeper's name is canonical (CONTEXT.md locked)
  - Extract typed value via `extractTypedValue()`
  - If keeper has the same propertyId:
    - Compare human-readable values. If equal: skip silently (not a conflict)
    - If different: add to `conflicts[]` with both values
  - If keeper does NOT have the propertyId:
    - Add to `propertiesToTransfer[]` with the extracted TypedValue
  - **Description property** (`SystemIds.DESCRIPTION_PROPERTY`): same logic -- if keeper has description, it's a conflict; if not, transfer it

**Relation comparison** (CONTEXT.md locked decisions):
- Build keeper relation dedup set via `buildKeeperRelationSet()`
- **Outgoing relations from merger:**
  - For each merger outgoing relation:
    - Skip if relation is a type assignment (typeId === SystemIds.TYPES_PROPERTY) -- handled separately in type comparison
    - Build dedup key: `out:${rel.toEntity.id}:${rel.typeId}`
    - If keeper already has this key: add to `relationsSkipped[]`
    - If keeper does NOT have this key: add to `relationsToRepoint[]` with `direction: 'outgoing'`
- **Incoming relations (backlinks) on merger:**
  - For each merger backlink:
    - Skip backlinks FROM the keeper entity itself (self-referential after merge would be odd)
    - Build dedup key: `in:${bl.fromEntity.id}:${bl.typeId}`
    - If keeper already has this key: add to `relationsSkipped[]`
    - If keeper does NOT have this key: add to `relationsToRepoint[]` with `direction: 'incoming'`

**Type comparison** (CONTEXT.md: union -- add merger's types that keeper doesn't have):
- `typesToTransfer = merger.typeIds.filter(t => !keeper.typeIds.includes(t))`

**Merger deletion ops:**
- Call `buildDeleteOps([merger])` from delete-builder.ts to get the ops that blank the merger entity
- Store as `mergerDeleteOps: deleteResult.ops`

Return the complete `MergePairDiff`.

**4. `buildMergeOps(diff: MergePairDiff, keeperId: string): Op[]`**

Convert a MergePairDiff into an Op[] array for a single atomic publish:

1. **Property transfer ops:**
   - Separate description from regular properties (same pattern as update command's publish phase)
   - Find description transfer: `diff.propertiesToTransfer.find(p => p.propertyId === SystemIds.DESCRIPTION_PROPERTY)`
   - Build values array from remaining transfers: `{ property: p.propertyId, ...p.typedValue }`
   - If any values or description: push `Graph.updateEntity({ id: keeperId, values, description }).ops`

2. **Type assignment ops:**
   - For each typeId in `diff.typesToTransfer`:
     - Push `Graph.createRelation({ fromEntity: keeperId, toEntity: typeId, type: SystemIds.TYPES_PROPERTY }).ops`

3. **Relation re-pointing ops** (delete old + create new):
   - For each relation in `diff.relationsToRepoint`:
     - Push `Graph.deleteRelation({ id: rel.relationId }).ops`
     - If `direction === 'outgoing'`: push `Graph.createRelation({ fromEntity: keeperId, toEntity: rel.otherEntityId, type: rel.typeId }).ops`
     - If `direction === 'incoming'`: push `Graph.createRelation({ fromEntity: rel.otherEntityId, toEntity: keeperId, type: rel.typeId }).ops`
   - CRITICAL: For incoming (backlinks), `fromEntity` is the third-party entity (otherEntityId), NOT the keeper. See RESEARCH.md Pitfall 5.

4. **Merger deletion ops:**
   - Push all ops from `diff.mergerDeleteOps`

Return the flat Op[] array.
